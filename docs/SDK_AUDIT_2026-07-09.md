# DTF SDK Audit and Register Integration Plan

Audited 2026-07-09 against the SDK monorepo, Register, reserve-api, their installed package versions, tests, and production workflow history.

## Executive result

The SDK architecture is healthy and should remain product-shaped: explicit namespaces/refs, a configured client, small source-specific transports, deterministic mappers, and prepared calls that stop at the wallet boundary. The code does not need a framework rewrite or generic hook factory.

The main ecosystem problem is adoption lag, not a missing foundation. Register is still locked to `@reserve-protocol/react-sdk` 0.2.0 while this repo is on 0.3.2 before the changes in this audit. That leaves shipped SDK reads/hooks unused and keeps duplicate API, subgraph, wagmi, and calldata logic alive in Register.

This audit closed three immediate SDK gaps:

- Index brand metadata now preserves `video` and downloadable `files`, removing the reason for Register's second folio-manager request.
- Ten existing Index core reads now have canonical React Query keys, query options, and hooks: mandate, total supply/assets, holders, bids-enabled, rebalance control/detail, active/latest auction, and bid quote.
- Query options now use TanStack Query's native `queryOptions()` branding and work directly in `useQueries`; Register no longer needs to rebuild selector-registry keys/functions by hand.

## Architecture

### What is working

- Core/wallet separation is correct: reads and `ContractCall`/`ContractCallPlan` preparation stay in the SDK; apps retain simulation, signing, submission, notifications, and refresh behavior.
- Namespace and ref APIs are explicit and flat. The domain factories in Index governance/vote-lock and the Yield namespace are easier to change than a generic binder would be.
- Source ownership remains visible. RPC, subgraph, Reserve API, catalog, and explorer reads are not hidden behind a generic resolver.
- On-chain integer values consistently use bigint/`Amount`; display-class ratios, timestamps, counts, rates, and subgraph BigDecimal analytics use numbers.
- Generated GraphQL and ABI files account for most apparent repository size; authored non-test/non-generated/non-ABI SDK + React source is about 20.6K lines.

### Maintainability findings

1. `packages/react-sdk/src/yield-dtf-hooks.ts` is 745 lines and mixes reads, query options, and call hooks across issuance, staking, governance, auctions, and APY. Split it by those real domains when the next Yield hook lands; this is maintainability work, not a performance emergency.
2. The root SDK export file is large because the supported design exposes the full ABI/helper surface. Keep the one-stop package, but continue composing domain files below it. Do not add a second public export hierarchy without measured consumer need.
3. Several mature core read files lack focused failure/mapping tests even though their downstream builders are well tested. Add tests when those reads change rather than chasing coverage percentage.
4. GraphQL freshness was not previously enforced locally. The workflow now maps GraphQL/codegen changes to a generated-output diff check.
5. Two existing lint warnings used ambiguous `new Array(length)` construction; both were replaced with explicit `Array.from`.

## Test quality

Current targeted totals after this audit:

- Core SDK: 260 passing unit tests; 17 opt-in live checks are skipped in the normal suite.
- React SDK: 71 passing tests.
- Agent workflow: 24 passing tests.

Strongest areas:

- contract-call shape and runtime validation;
- governance/vote/rebalance/deploy call encoding;
- Index proposal mapping/state and optimistic governance;
- Yield proposal/call encoding;
- query-key normalization and stale-time behavior;
- transport failure context.

Next test investments, in value order:

1. Add exact-byte golden fixtures for the highest-value assembled Index basket/settings proposals. Existing decode-roundtrip tests are good behavior checks, but golden bytes detect ABI/selector drift.
2. Add React provider/identity tests and one mocked end-to-end query hook; most React tests currently prove options/call construction rather than provider/query integration.
3. Add mapper fixtures when extending a response shape, as done for brand `video/files`; do not add compiler-duplicate tests.
4. Keep live checks opt-in, but run Index and Yield smoke suites before releases that change RPC/subgraph/API boundaries.

## React SDK shape and performance

The current shape is right:

- one root provider and one optional route identity provider;
- plain `useQuery` wrappers over exported query options;
- normalized keys with chain identity and bigint/address normalization;
- explicit 10s/30s/5m freshness classes;
- `select` support without atom mirroring;
- no hook factory or runtime registration layer;
- core SDK, React, TanStack Query, and viem externalized from the published React wrapper.

Measured production bundles with Vite 8, retaining exports so tree-shaking could not erase them:

| Entry                                                |    Minified |      Gzip |
| ---------------------------------------------------- | ----------: | --------: |
| React + TanStack Query baseline                      |    20.88 kB |   7.36 kB |
| One core `getIndexDtfPrice` import                   |   358.45 kB |  62.72 kB |
| One `useIndexDtfPrice` import                        |   385.29 kB |  71.31 kB |
| Namespace import of every React/core export          | 1,015.88 kB | 213.03 kB |
| Published React wrapper itself (core/peers external) |    70.76 kB |   8.17 kB |
| Price reader after preserve-modules packaging        |    15.41 kB |   5.12 kB |

Named imports originally tree-shook roughly two thirds of the full one-stop surface, but a follow-up module trace found real packaging leakage. In a direct `getIndexDtfPrice` consumer build, rendered module weight was approximately 49% the SDK bundle, 21% Zod, 12% viem, 8% `dtf-rebalance-lib`, and 4% `decimal.js-light`. The React wrapper itself was small; the issue was the monolithic core distribution entry and its top-level schema/dependency initialization, not hook overhead or wagmi.

The SDK now publishes preserve-modules output while keeping the same root entry. In the direct price-reader trace, the bundle fell from 538.08 kB to 15.41 kB and no longer retains Zod, `dtf-rebalance-lib`, or `decimal.js-light`; the remaining weight is principally viem address/encoding code. `check:sdk-bundle` enforces a 25 kB/8 kB-gzip budget and asserts those unrelated dependencies stay absent. Register already carries React, TanStack Query, and viem, so its incremental cost is smaller still. Avoid `import * as sdk` in browser entry points.

Provider performance depends on stable configuration identity. Passing a newly created `chains` object every render recreates the SDK and rerenders all SDK-context consumers. Apps should define configuration once or pass a pre-created `sdk` instance; the React README now states this explicitly.

## Documentation audit

The documentation structure is strong: canonical protocol/source docs, a separate public Vocs site, explicit plans, and an LLM guide. Drift existed at the entry points:

- the root README still claimed Yield methods were unimplemented;
- the catalog README still claimed the published package was private;
- no React package README existed;
- the old improvement plan mixed completed release history, stale package pins/test totals, and current backlog;
- the Register migration plan still named removed `getFull` API and listed hooks that have since shipped.

Entry-point claims were corrected in this pass. Historical documents remain useful evidence, but this file owns the current audit and integration order. Plans must be checked against source and the consumer's installed version before execution.

## Register integration plan

Register currently has 51 SDK imports but 78 files using React Query and many direct Index/Yield RPC, subgraph, API, and calldata paths. Migrate one coherent product flow at a time; do not big-bang replace atoms.

### P0 — release and version alignment

1. Release SDK + React SDK together with Changesets, then deliberately bump Register from 0.2.0. A caret on 0.2.0 does not accept 0.3.x.
2. Link/build both packages during migration so React does not resolve a stale core SDK instance.
3. Run Register typecheck/tests plus manual smoke on one Ethereum and one Base DTF before deleting old data paths.

### P1 — delete duplicate Index route reads

Migrate these together because they live in `index-dtf-container.tsx` and currently mirror query data into route atoms:

- delete `BrandFilesUpdater`; `useCurrentIndexDtf().data.brand` now includes `video/files`;
- add `platformFee` to the full/current DTF route model and delete `PlatformFeeUpdater`; retain `useIndexDtfPlatformFee` for focused consumers that should not pay for the full DTF read;
- replace the manual exposure request with `useIndexDtfExposure` and keep only the Register-specific atom adapter until consumers move to hooks;
- add `status` to the full/current DTF route model and delete the route status updater; source it from shared/batched discovery rather than one paginated status scan per DTF, and keep `KNOWN_DEPRECATED` until the aggregate has equivalent instant fail-safe behavior;
- use `useIndexDtfPrice`/`useIndexDtfPriceHistory` for overview price sources;
- remove redundant atom writes consumer-by-consumer, leaving route UI to select/query/render.

Exit evidence: no second folio-manager request; one query key per source; brand cover does not flap; exposure/status/platform fee match production values.

### P2 — auctions, settings, and manual issuance

- Replace auction/rebalance subgraph hooks with `useIndexDtfRebalances`, `useIndexDtfRebalance`, `useIndexDtfRebalanceAuctions`, `useIndexDtfCurrentRebalance`, `useIndexDtfActiveAuction`, and `useIndexDtfLatestAuction`.
- Replace live bid calculation reads with `useIndexDtfBidQuote`; Register retains form state and transaction submission.
- Replace propose/settings wagmi reads with `useIndexDtfBidsEnabled` and `useIndexDtfRebalanceControl`.
- Replace supply/assets and holder reads with `useIndexDtfTotalSupply`, `useIndexDtfTotalAssets`, and `useIndexDtfHolders`.
- Move the shared rebalance-health boundary into the SDK: the Reserve API `/rebalance/liquidity` client and response types, matched trade-size derivation, Ondo capacity/trading-state mapping, scaled leg sizes, and max-safe-percent/exceeded-leg calculations. Reuse the existing SDK current-rebalance and open-auction builders instead of Register's duplicate version transforms and `getOpenAuction` plumbing.
- Keep debounce timing, per-token retry UI, warnings/copy, atom/form state, and Zapper token-support checks in Register. Those are interaction behavior or a separate integration boundary, not protocol calculation.
- Use `useIndexDtfIssuanceState` for manual mint/redeem wallet state; keep Zapper/async-mint orchestration in their dedicated SDKs until a stable Zapper HTTP contract is explicitly adopted.

### P3 — Yield migration

Execute the existing wave order in `REGISTER_YIELD_IMPLEMENTATION.mdx`: staking, auctions, issuance, governance, overview, then settings. The first four waves already have substantial SDK coverage.

Five core reads still block complete Yield cleanup and should be built consumption-first:

1. `getConfiguration` — governed parameters used by settings/proposal forms.
2. `getAssets` — registry assets, prices, max trade volumes, and versions.
3. `getBackupBaskets` — target-unit backup configuration.
4. collateral status reads — SOUND/IFFY/DISABLED basket health.
5. `getSupplyHistory` — hourly/daily subgraph snapshots for charts.

Each new read gets a namespace/ref method, query key/options/hook, a frozen mapper/contract fixture, and one live validation case before its Register consumer is removed.

### P4 — remaining product boundaries

- Keep recent just-submitted proposals in Register because transaction receipts and optimistic UI are application state.
- Keep wallet, compliance, route, notification, and transaction lifecycle code in Register.
- Keep Zapper and wrapper-specific Yield issuance in their existing dedicated libraries unless a concrete non-Register consumer needs a core HTTP wrapper.
- Keep reserve-api dependent on the catalog, not the client SDK. reserve-api implements the HTTP source the SDK consumes; importing the client SDK back into it would create the wrong dependency direction. Share only concrete schemas if drift becomes a measured problem.

## Release checklist for this audit

- Changeset added for SDK + React SDK: brand response expansion, new hooks/query options, and `useQueries` compatibility.
- Full uncached workflow gate passed under Node 24: builds, types, lint/format, 331 unit tests, docs site/routes/links, and catalog invariants.
- `pnpm release:ci` passed, including dry-run tarballs for all three public packages.
- Independent correctness/security/product/complexity review is clean. Human review remains required before publishing the public API additions and query cache-key shapes.
