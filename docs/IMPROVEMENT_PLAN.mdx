# DTF SDK Improvement Plan

Audit date: 2026-06-11. Scope: `@reserve-protocol/sdk`, `@reserve-protocol/react-sdk`, `@reserve-protocol/dtf-catalog`, plus how Register (`~/projects/register`, react-sdk `^0.1.5`, ~48 files) consumes them. Dimensions: code quality, patterns, performance, security.

Method: parallel codebase audits per package + a consumption audit of Register, with every load-bearing finding hand-verified against source. Findings that didn't survive verification are listed at the bottom so future audits don't re-flag them.

## Product framing

These decisions drive every priority below:

- **Register is the main consumer and shapes the SDK**, but the SDK is becoming the main tool for the broader stack (dtf-ai, bots, custom pages). Optimize for completeness and correctness, not surface minimization.
- **Breaking changes are fine.** SDK and Register ship together in lockstep. Every breaking item below includes the Register migration as part of the same work item — no deprecation shims, design the best API and fix Register with it.
- **Register's partial adoption is intentional.** Unused hooks are future surface, not dead code. Don't migrate Register to features it doesn't need yet.
- **react-sdk is the one-stop import.** `export * from "@reserve-protocol/sdk"` stays. Everything SDK-related — including ABIs — must be reachable through it.
- **dtf-catalog stays plain JSON.** No validation layer.
- **Testing must be rock solid** — it's a top-level pillar (P1), not hygiene.

Severity legend: **High** = correctness/money risk or major perf impact. **Medium** = maintenance/consistency cost that compounds. **Low** = note it, fix opportunistically.

## Package: @reserve-protocol/sdk

Structure: 115 source files / 29 test files. Domains: `index-dtf` (dtf, governance, rebalance, vote-lock, deploy), `client` (viem/subgraph/api/explorer transports), `lib`, `types`. ESM-only via tsdown, `sideEffects: false`. Minimal deps (viem, graphql-request, decimal.js-light, zod) — all justified.

### Code quality

**[High] `as never` cast kills type safety on all calldata encoding** — `src/lib/contract-call.ts:67`

```ts
const data = encodeFunctionData({
  abi: params.abi,
  functionName: params.functionName,
  args: params.args,
} as never) as Hex;
```

Every governance write (vote, queue, execute, cancel, propose) encodes through this function. The `as never` suppresses viem's ABI/functionName/args type checking, so a wrong arg shape compiles fine and produces malformed calldata at runtime. This is the single highest-stakes line in the SDK. Fix: make the generics line up so `encodeFunctionData` typechecks for real, or constrain the param types until the cast is unnecessary.

**[High] ABI export gap** — `src/index.ts`

~28 ABI consts are defined in the SDK but only 8 are exported from the root: `dtfIndexAbi`, `dtfIndexGovernanceAbi`, `dtfIndexGovernanceOptimisticAbi`, `dtfIndexGovernanceProposalAbi`, `dtfIndexStakingVaultAbi`, `dtfIndexStakingVaultOptimisticAbi`, `indexDtfV5WriteAbi`, `indexDtfV6WriteAbi`.

Missing from the root export: `timelockAbi`, `optimisticTimelockAbi`, `selectorRegistryAbi`, `unstakingManagerAbi`, `indexDtfDeployerAbi`, `indexDtfGovernanceDeployerAbi`, `daoFeeRegistryAbi`, `dtfIndexProposalAbi` + the version catalog (`dtfIndexProposalAbiCatalog`, v1/v2/v4/v5 proposal ABIs), `dtfAdminProposalAbi`, `upgradeSpellProposalAbi`, `folioArtifactAbi`, `indexDtfProposalDecoderAbi`, `dtfIndexAbiV1/V2/V4`.

Consumers (Register, bots, custom pages) end up copy-pasting ABIs the SDK already has. Exporting them from the sdk root automatically exposes them through react-sdk's `export *`.

**[Medium] `Number()` on subgraph money strings** — multiple mappers

- `src/index-dtf/dtf/mappers.ts` (timestamps, fees, revenue totals)
- `src/index-dtf/governance/mapper.ts` (delegate vote counts)
- `src/index-dtf/governance/voting.ts:79` (`Number(votingPower) / Number(voteSupply)` — acceptable as a display percentage, but the inputs are raw vote weights)
- `src/index-dtf/dtf/transactions.ts:56` (`Number(amount.formatted) * dtfPriceUsd`)
- react-sdk `src/index-dtf/index-dtf-data.ts:92-93` (`Number(dtf.fees.mintingFee.formatted)`)

JS numbers lose precision past 2^53 and on long decimal strings. Some of these are display-class values (percentages, timestamps) where `number` is the right call; others are money. The fix is a triage, not a blanket conversion — see the money rule in Direction Feedback. **Any type change ships with the Register call sites updated in the same release** (e.g. `mintingFee`/`tvlFee` consumers of `mapIndexDtfData`).

**[Low] Duplicate dedupe utilities** — `src/lib/utils.ts` has both `uniqueAddresses` and `dedupeAddresses` doing the same job. Keep one.

**[Low] One-off lowercase address comparison** — `src/index-dtf/governance/proposals.ts:342` compares with manual `.toLowerCase()` while the rest of the codebase normalizes with `getAddress`. Harmless, just inconsistent.

### Patterns

**[Medium] `index.get` and `index.getFull` are the same function** — `src/index-dtf/namespace.ts:108-110`

```ts
get: (params: GetIndexDtfParams) => getFull(client, params),
getFull: (params: GetFullIndexDtfParams) => getFull(client, params),
```

react-sdk mirrors the duplication: `indexDtfQueryOptions` and `fullIndexDtfQueryOptions` (`src/query-options.ts:73-97`) hit the same query key and the same underlying function. Two names for one thing is API ambiguity that every new consumer pays for. Breaking-OK: pick one name (`get` reads better as the flat API), delete the other across sdk + react-sdk, update Register if it touches the removed name.

**[Medium] Transaction mapper derives business state** — `src/index-dtf/dtf/transactions.ts:56`

The mapper accepts `dtfPriceUsd` and computes `amountUsd`. AGENTS.md says mappers are deterministic raw-shape conversion only. Move the USD derivation to the caller (or a dedicated enrichment step) and keep the mapper pure.

**[Medium] Governance domain outgrew its structure** — `src/index-dtf/governance/`

7+ files (proposals, voting, delegates, decoder, optimistic, guardians, proposal-actions, propose/) with flat wildcard barrels, while vote-lock has the proper `reads.ts`/`builders.ts`/`namespace.ts`/`ref.ts` shape AGENTS.md mandates for grown domains. Restructure for navigability. Note: per the "expose everything" direction, this is about file organization, not hiding exports — the public surface stays complete.

### Performance

**[Medium] Sequential page polling in discovery** — `src/index-dtf/dtf/discovery.ts:145-160` fetches discovery pages one at a time in a while loop. Fine for small catalogs, slow at scale. Low urgency until DTF count grows; flagging so it's a known ceiling.

**[Low] Default 1000-item cap on subgraph discovery** — `src/index-dtf/dtf/discovery.ts:113`. Document the cap or expose pagination; silent truncation is the failure mode to avoid.

Otherwise good: multicall batching is used in the read-heavy paths (vote-lock, rebalance), `Promise.all` where it matters, Decimal.js for basket math.

### Security

Clean. No secrets or API keys committed; subgraph URLs in `src/config/index.ts` are public Goldsky endpoints. Addresses normalized through `getAddress` consistently. GraphQL variables go through `graphql-request` against a typed schema. The `as never` finding above is the only thing in the codebase with real funds-adjacent risk, and it's a type-safety issue, not an exploit.

### Tests

29 test files for 115 source files, concentrated on reads and mappers. Zero coverage on: `lib/utils.ts`, `lib/contract-call.ts` (calldata encoding!), `transports/` error paths, `client/api`. The live smoke test covers Index DTF reads only. See P1.

## Package: @reserve-protocol/react-sdk

Structure: ~60 hooks + ~50 query option builders over the core SDK, two-tier provider (`DtfSdkProvider` → `IndexDtfProvider`), centralized query keys with bigint/address normalization. Peer deps: react 18/19, @tanstack/react-query 5, viem 2. 4 test files (~820 lines), all on query keys/options.

### Performance

**[High] No `staleTime`/`gcTime` anywhere** — verified by grep across the package

Every query uses React Query's default `staleTime: 0`, so every mount refetches. Register renders governance pages with a dozen SDK queries; all of them re-hit the subgraph/API on every navigation. This is the highest-impact performance fix in the repo and it's additive (defaults remain overridable through the existing `options` param every builder already accepts).

Suggested defaults by data class, set inside `createDtfQueryOptions` callers:

| Data class          | Examples                                   | staleTime |
| ------------------- | ------------------------------------------ | --------- |
| Prices              | `indexDtfPriceQueryOptions`, price history | ~10s      |
| Governance state    | proposals, voter state, vote-lock          | 30–60s    |
| Static-ish metadata | DTF identity, brand, version, catalog      | ~5min     |

### Code quality

**[Medium] Type helpers copy-pasted in 3 files** — `IndexMethod`/`PortfolioMethod`/`MethodParams`/`MethodResult` are defined identically in `src/query-keys.ts:36-43`, `src/index-dtf-query-options.ts:6-13`, and `src/index-dtf-extra-hooks.ts:28-35`. Move to one shared module.

**[Medium] Duplicate query option builders** — `indexDtfQueryOptions` vs `fullIndexDtfQueryOptions` (see the `get`/`getFull` finding above; one removal fixes both layers).

**[Low] `normalizeQueryKeyValue` doesn't normalize `number`** — `src/normalize-query-key.ts`. bigints become strings but numbers pass through, so `100` and `100n` produce different cache keys for the same value. Low risk since the SDK uses bigint for on-chain values, but cheap to close.

### Patterns

The query key architecture is good: deterministic normalization, address lowercasing, sorted governance address sets, and `enabled` is computed internally from `params !== undefined` (`src/query.ts:30`) so hooks accept `undefined` params safely — consumers don't need their own guards. Action hooks (`useIndexDtfVoteCall` etc., `src/index-dtf/use-index-dtf-proposal-actions.ts`) return prepared calls via `useMemo` rather than queries; the repeated param extraction across the four hooks is deliberate explicit style per AGENTS.md, not a defect.

### Tests

Query keys and option builders are well covered. The 60+ hooks have zero tests — including the four proposal action hooks that build transaction calls. See P1.

## Package: @reserve-protocol/dtf-catalog

Plain JSON catalogs (index + yield DTFs for chains 1, 56, 8453) with thin typed wrappers. **Plain on purpose** — no validation layer wanted, data is manually curated.

One note worth a comment in code, not a fix: the merged `dtfs` record (`src/index.ts:22-34`) spreads yield then index, so an address present in both silently resolves to the index entry. If that precedence is intentional (it appears to be), say so in a one-line comment.

## Register consumption audit

### Usage inventory

Register uses **~18 of the ~32 governance-area hooks** plus `dtfQueryKeys`, both providers, and a handful of types (`IndexDtfProposalSummary`, `Amount`, `SupportedChainId`, `IndexDtfData`). Heaviest: `useIndexDtfIdentity` (11 sites), `useIndexDtfProposal` (10), `useDtfSdk` (9), `dtfQueryKeys` (9).

Unused surface — **intentionally**, per the partial-adoption strategy: the optimistic-governance hook family, selector-registry hooks, and all 23 "extra" hooks (rebalance, portfolio, price, status, revenue). These are future surface for dtf-ai, bots, and custom pages. Not deprecation candidates.

### Gaps the SDK should absorb (breaking-OK, ship with Register migration)

**Vote-lock state should include token prices.** Register's `src/components/vote-lock/hooks/use-fallback-vote-lock-state.ts` (~120 lines) exists because the SDK's vote-lock state lacks underlying token prices — it manually calls `sdk.client.api.getTokenPrices()` and assembles the drawer state. Add prices to the SDK's vote-lock state read (or a sibling read that includes them), then collapse Register's fallback hook to the SDK hook.

**Proposal + voting snapshot should be one read.** Register's `withVotingSnapshot()` in `src/views/index-dtf/governance/views/proposal/hooks/use-proposal-detail.ts:34-77` manually merges `useIndexDtfProposal` with `useIndexDtfProposalVotingSnapshot`, including threshold-keeping logic. That merge is generic, not Register-specific — absorb it as an SDK read/hook and delete the local merge.

### Register-side consumption fixes (no SDK change required)

- `src/views/index-dtf/governance/views/propose/shared/hooks/use-proposal-type-eligibility.ts:82-84` calls `sdk.index.getSelectorRegistryIsAllowed` raw inside `useQueries` — `indexDtfSelectorRegistryIsAllowedQueryOptions` already exists and composes with `useQueries` directly. Switching gets cache keys and enabled handling for free.
- `src/components/vote-lock/hooks/use-vote-lock-refresh.ts` hand-manages `dtfQueryKeys` invalidation — re-verify the keys after the vote-lock changes above; it likely shrinks.
- `mapProposalDetail()` in `use-proposal-detail.ts:79-131` re-shapes the SDK's proposal detail — revisit after the snapshot merge lands; some of it should disappear.

### Intentional frontend boundaries (not SDK gaps)

`src/views/index-dtf/governance/utils/recent-proposals.ts` (~456 lines) parses `ProposalCreated` events from tx receipts to show proposals before the subgraph indexes them. This **stays in Register**: it requires the receipt from the just-submitted transaction, which only the frontend has. The SDK's contract is subgraph-indexed data; this is the documented seam between the two.

### Migration state

Governance is fully on the SDK (proposals, voting, voter state, proposal actions, vote-lock with the fallback noted above). Rebalance, portfolio, and price surfaces are not migrated — by choice, pending feature work. Treat migration % as a strategy artifact, not a debt metric.

## Direction feedback

Asked for, so here it is straight:

**Lockstep breaking releases are the right call today — with one cheap hedge.** While every consumer is yours (Register, dtf-ai, bots), shipping SDK + consumers together is strictly faster than maintaining deprecation paths. The door closes the moment something external pins a version. The hedge: keep changeset notes honest about what broke and how to migrate, even for internal releases. It costs minutes now and becomes the migration guide the day the SDK has a consumer you don't control.

**State the money rule once and enforce it in mappers.** The Number findings aren't 15 separate bugs, they're one missing convention: anything that is money is `Amount { raw: bigint, formatted: string }`; plain `number` is reserved for display-class values (percentages, timestamps, counts). Mappers enforce it at the boundary and everything downstream inherits it. One sentence in AGENTS.md, applied in P2 item 10, done.

**Calldata encoding is the testing center of gravity.** A wrong subgraph mapping shows a wrong number on a screen; wrong calldata queues a wrong governance action on-chain. That asymmetry should shape the test suite — exact-encoding fixture tests for every prepare/build function matter more than any coverage percentage. P1 item 4 is the most valuable test work in the repo.

**The one-stop-import design is right; ambiguity is the actual enemy.** Re-exporting the whole SDK through react-sdk is good DX for a Register-first SDK, and `sideEffects: false` keeps bundlers happy. The thing to guard is two names for one thing (`get`/`getFull`) and defined-but-not-exported artifacts (the ABIs). Completeness plus exactly-one-way-to-do-it is what makes the SDK feel like "the main tool" to a bot author or the dtf-ai project.

## Non-findings & intentional choices

Checked and fine — don't re-litigate in future audits:

- **Security**: no secrets/keys committed; public Goldsky subgraph URLs only; consistent `getAddress` normalization; schema-typed GraphQL variables.
- **`export * from "@reserve-protocol/sdk"` in react-sdk**: by design — react-sdk is the one-stop import.
- **dtf-catalog as plain JSON**: by design, manually curated.
- **Register partial adoption**: deliberate strategy — migrate as features change.
- **`recent-proposals.ts` in Register**: inherently frontend (needs the tx receipt); not an SDK gap.
- **Action hook param extraction repetition**: intentional explicit style per AGENTS.md.

Retracted findings from the audit (verified false):

- _"Register hooks missing `enabled` guards"_ — the SDK computes `enabled` from `params !== undefined` internally (`react-sdk/src/query.ts:30`); passing `undefined` params is the intended usage.
- _"Governance decoder silently returns partial decodes"_ — `unknownCalls` on the result is the contract; Register intentionally renders unknown calls as raw calldata.
- _"Duplicate query options cause cache collisions"_ — `indexDtfQueryOptions` and `fullIndexDtfQueryOptions` share a key but also share the underlying function, so the cache is coherent; the problem is duplication (P2 #9), not corruption.
