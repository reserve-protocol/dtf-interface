# DTF SDK Improvement Plan

Audited 2026-06-11; the P0–P3 roadmap from that audit was **executed the same day** on branch `chore/sdk-audit-improvements` (SDK) with the matching Register migration left uncommitted in `~/projects/register`. This document now records: what shipped, what the audit got wrong (verified during execution), and the next plan.

## Product framing (unchanged — drives all priorities)

- **Register is the main consumer and shapes the SDK**, but the SDK is becoming the main tool for dtf-ai, bots, and custom pages. Optimize for completeness and correctness.
- **Breaking changes are fine.** SDK and Register ship together in lockstep; no deprecation shims.
- **Register's partial adoption is intentional.** Unused hooks are future surface, not dead code.
- **react-sdk is the one-stop import** — `export * from "@reserve-protocol/sdk"` stays; everything (including ABIs) reachable through it.
- **dtf-catalog stays plain JSON.**
- **Money rule** (now in AGENTS.md): on-chain integer amounts are always `Amount { raw, formatted }` via `mapAmount`; plain `number` is reserved for display-class values (counts, timestamps, durations, percentages, rates, subgraph BigDecimal analytics).
- **Testing is rock solid as a pillar** — calldata encoding is the center of gravity.

## Corrected findings (the audit was wrong; don't re-flag)

1. **"Governance domain lacks namespace/ref structure"** — false. `governance/namespace.ts` and `governance/ref.ts` exist and compose into the root factories. Topical read files (proposals/voting/delegates/…) beat a monolithic `reads.ts`; no restructure done or needed.
2. **"Money paths use Number()"** — mostly false. Everything weight/amount-like already maps through `mapAmount`; the flagged `Number()` calls are counts (`forDelegateVotes` is a delegate count, not weight), timestamps, durations, and display percentages. Revenue fields are subgraph BigDecimal analytics — display-class by design (NOTE comments added at the two ambiguous spots; rule added to AGENTS.md).
3. **"Vote-lock state lacks prices"** — stale. `getVoteLockState` already returned `underlyingPrice`. The real gap was the DTF-address requirement; fixed with the vault-keyed read.
4. **"Zero transport/API tests"** — overstated. `transports/subgraph` had 5 tests including error context; `client/api` had happy-path coverage. The real gaps (api error paths, contract-call, utils) are now filled.
5. **"as never produces silent malformed calldata"** — overstated. The public signature always type-checked callers, and viem throws at runtime on arg mismatches (now proven by a test). The fix still landed for honesty of the internal types.

## Intentional choices (don't re-litigate)

Security clean (no secrets, public Goldsky URLs, consistent `getAddress`). `export *` re-export is the design. dtf-catalog plain JSON, index-over-yield merge precedence now commented in code. `recent-proposals.ts` stays in Register (needs the tx receipt — inherently frontend). Proposal action hooks' repeated param extraction is deliberate explicit style. Register `mapProposalDetail` is Register's own legacy boundary — shrink only when its consumers change.

## Next plan

Ordered by value; nothing here blocks the release of what's above.

### Release (do first)

1. **Changesets + lockstep release.** One changeset covering the breaking changes (getFull removal, getTransactions param, vote-lock additions). Publish sdk + react-sdk, bump Register's `@reserve-protocol/react-sdk`, commit Register's migration files (4 files + 1 deletion), ship together. Register's vote-lock and governance flows should get a manual smoke pass (`/verify`) after the bump.

### Quality (next sessions)

2. **Hook test coverage expansion** (S–M): the jsdom harness now exists; add tests for `useIndexDtfProposalList` select behavior, `IndexDtfProvider` identity, and one query-flow hook with a mocked sdk. Target the hooks Register leans on hardest (`useIndexDtfIdentity`, `useIndexDtfProposal`).
3. **Propose-builder calldata fixtures** (M): `buildBasketProposal`/`buildSettingsProposal` produce governance calldata; they have behavior tests but not exact-encoding fixtures like `castVote` now has. Same pattern, highest-stakes remaining code.
4. **Discovery pagination ceiling** (S): `discovery.ts` still polls pages sequentially and defaults to a 1000-item subgraph cap. Fine today; document the cap in the JSDoc now, parallelize when DTF count makes it visible.

### API health (opportunistic)

5. **`useQueries` interop** (S): RQ v5's `UseQueryOptionsForUseQueries` rejects `DtfQueryOptionsResult` (branded `queryFn`/`placeholderData` variance). If more multi-query consumers appear, export a `forUseQueries()`-shaped builder or adopt RQ's `queryOptions()` branding in `createDtfQueryOptions`.
6. **Register `use-vote-lock-refresh.ts`** (S): still hand-manages invalidation keys; now that the fallback is an SDK hook with a canonical key, the external-refresh branch can likely invalidate `dtfQueryKeys.index.voteLockVaultState(...)` instead of threading refetch callbacks. Do it when next touching the drawer.
7. **`sdk.yield` boundary** (done): the placeholder namespace has been replaced with the implemented Yield DTF surface. Keep repeating the calldata-fixture pattern as new Yield builders land.

### Watchlist

- The `Number(bigint)` ratio in `votingWeight` rounds above 2^53 — harmless for a percentage, but if anyone ever sorts/compares by it for consensus-critical UI, switch to a Decimal ratio.
- `normalizeQueryKeyValue` doesn't normalize `number` vs `bigint` for the same value — irrelevant while params are typed, cheap to close if a cache-miss bug ever points there.

# Yield DTF SDK

Phase 1 shipped 2026-06-12 on `feat/yield-dtf-phase-1`. Protocol architecture reference: `docs/yield-dtf/PROTOCOL.md`. Behavioral source of truth: Register's `src/state/rtoken/*` and `src/views/yield-dtf/*`.

## Phase 1 — shipped

- ABIs (facade-read/act, r-token, main, st-rsr, st-rsr-votes, basket-handler), facade + RSR addresses for 1/8453, `YieldDtfChainId`.
- Subgraph codegen target against the deployed `dtf-yield` Goldsky subgraph + phase-1 queries.
- Core reads: `get` (subgraph identity + on-chain mandate/basket tokens), `list`, `getContracts` (Main components), `getState` (supply/throttles/pauses — protocol >= 3.0 only), `getPrice` (facade), `getBasket` (breakdown/backing/buffer), `getHolders`, `getTransactions`.

- `createYieldDtfNamespace` + `createYieldDtfRef`; `sdk.yield` placeholder replaced.
- react-sdk: `dtfQueryKeys.yield.*`, 23 query option builders + hooks with staleTime classes, 13 selected call hooks.
- Tests: exact-calldata fixtures for every prepare builder, frozen subgraph mapper fixtures, key/staleTime/call-hook tests, live smoke against mainnet eUSD (all 4 passing live).

## Phases 2-4 — SHIPPED (2026-06-12, same branch)

All three remaining phases executed with per-phase adversarial reviews and live validation:

- **Phase 2 (governance)**: shared OZ governor builders extracted to `lib/governor-calls.ts` (index-dtf delegates to them, 120 index tests unchanged); yield governance reads with always-on-chain quorum/threshold (review caught: subgraph `quorumVotes` null for most Alexios frameworks — guaranteed crash — and `proposalThreshold` stored as micro-percent); authoritative on-chain proposal `state()`; voter state; both cancel paths (governor + guardian timelock via shared OZ operation-id math).

- **Phase 4 (APY + proposing)**: SDK-owned DefiLlama integration (pool map absorbed from Register's CollateralYieldUpdater), pure `computeYieldDtfApy` with golden-number tests, realized staking APY from exchange-rate snapshots; the full proposal-builder set (every governed parameter, roles, baskets, distribution) composing into the shared governor propose.
- **Validation**: `pnpm --filter @reserve-protocol/sdk validate:yield` — ~170 live checks against eUSD, ETH+ (mainnet), bsdETH (base), all passing. Workspace totals after phases: sdk 250+ tests, react-sdk 58 tests.

### Still open (feature-paced)

Five SDK prerequisite reads identified there (consumption-driven gap analysis, sweep over-claims corrected): `getConfiguration` (all governed params, HIGH), `getAssets` (AssetRegistry + prices/maxTradeVolume, HIGH), `getBackupBaskets` (MEDIUM), collateral `status()` reads (MEDIUM), `getSupplyHistory` (subgraph token snapshots for charts, MEDIUM).

## Known boundaries

- Protocol 2.x mains (legacy `paused()`) unsupported — all live listed Yield DTFs are >= 3.0.
- Arbitrum has deployed Yield DTFs but no configured subgraph in the SDK; add `42161` to `YieldDtfChainId` + subgraph URL when needed.
- `Collateral` subgraph entity has no decimals — `get` does an on-chain ERC20 metadata pass for basket tokens (one extra multicall).
