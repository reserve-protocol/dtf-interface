# Register × Yield DTF SDK — Integration Plan

Written 2026-06-12, after the full `sdk.yield` surface shipped (see `docs/yield-dtf/PROTOCOL.md`). This is the executable plan for migrating Register's yield-dtf data layer (9 atoms, 2 updaters, ~80 consumer components) onto the SDK, wave by wave, plus the SDK gaps that block specific waves. Migration is feature-paced per product direction — each wave ships when its view is next touched, no big bang.

## Ground rules

- **Lockstep releases**: SDK additions and the Register wave that consumes them land together.
- **`ContractCall` is wagmi-ready as-is**: every builder returns `{ to, data, value, contract: { abi, address, functionName, args } }` — spread `call.contract` straight into `useSimulateContract`/`writeContract`. No calldata-decoding adapter is needed (an earlier sweep suggested one; wrong).
- **Hooks over raw SDK**: views consume `useYieldDtf*` hooks; the SDK client comes from the existing `DtfSdkProvider` Register already mounts.
- **Atoms become thin**: where a jotai atom must survive (cross-view state), it wraps the hook result; data fetching itself moves to React Query via the SDK hooks.

## SDK prerequisites — real gaps, in priority order

These block specific waves. Each is a contained read in the established pattern (multicall, Amount mapping, fixture + live-smoke tests).

1. **`getConfiguration({ address, chainId, contracts? })`** — all ~20 governed params in one multicall across BackingManager/Broker/StRSR/Furnace/BasketHandler/RToken/Main: tradingDelay, backingBuffer, maxTradeSlippage, minTradeVolume, rewardRatio (furnace + stRSR), unstakingDelay, withdrawalLeak, warmupPeriod, issuancePremiumEnabled, batch/dutch auction lengths, throttle params, short/long freeze, reweightable. Replaces `rTokenConfigurationAtom`. Blocks: settings view, governance proposal forms. **HIGH**
2. **`getAssets({ address, chainId })`** — `AssetRegistry.getRegistry()` then per-asset multicall `price()`, `maxTradeVolume()`, `version()` + ERC20 meta. Replaces `rTokenAssetsAtom`. Blocks: issuance display details, governance asset proposals. **HIGH**
3. **`getBackupBaskets({ address, chainId })`** — target units from `basketBreakdown`, then `FacadeRead.backupConfig(rToken, targetName)` per unit. Replaces `rTokenBackupAtom`. Blocks: emergency-collateral view, backup-change proposals. **MEDIUM**
4. **Collateral status** — `status()` (0 SOUND / 1 IFFY / 2 DISABLED) per collateral asset; either extend `getBasket` or a `getCollateralStatus` read. Replaces `rTokenCollateralStatusAtom` feed. Blocks: basket health display. **MEDIUM**
5. **`getSupplyHistory({ address, chainId, interval, from })`** — the yield subgraph's `hourlyTokenSnapshot`/`dailyTokenSnapshot`; the SDK owns this subgraph, so charts shouldn't keep a parallel gql client. Replaces `supply-chart.tsx` inline queries. **MEDIUM**

### Wave 6 — Settings (after prerequisites 1–3)

`rTokenConfigurationAtom` → `getConfiguration`; `rTokenAssetsAtom` → `getAssets`; `rTokenBackupAtom` → `getBackupBaskets`. Settings is read-only display — pure swap.

## Stays in Register permanently

- **Zap + wrapping** (separate SDKs / wrapper-specific flows), **deploy** (FacadeWrite — its own future SDK phase if ever).
- **Chainlink ETH/BTC/RSR/gas price atoms** (UI-wide concerns, not yield-domain).
- **Chart rendering + collateral-detail enrichment** (`rTokenCollateralDetailedAtom` — combines SDK data with UI context).
- **Proposal form-diffing** (form state → which builders to call) and proposal simulation UX.
- **Batch (Gnosis) auction bidding** — SDK gap noted in IMPROVEMENT_PLAN; dutch covers current usage.

## Issues found in current SDK code during this analysis

None functional — the sweep's claimed gaps were either covered (see corrections above) or genuinely missing reads (the five prerequisites). Two sharp edges worth knowing during migration:

1. `getState` resolves Main components internally (3 RPC roundtrips); always pass the cached `contracts` param from a long-lived `useYieldDtfContracts` query in Register (STATIC staleTime) — this matches Register's current contracts-atom caching behavior.
2. `useYieldDtfProposalVotePower` keys include `voteStart`; for ACTIVE proposals where Register re-reads power per block, prefer a manual `refetchInterval` instead of key-busting.

## Verification per wave

Each wave: typecheck Register against the linked SDK build, run the affected view manually against eUSD (mainnet) + bsdETH (base), and diff displayed values against production Register side by side. SDK side: `pnpm --filter @reserve-protocol/sdk validate:yield` stays the regression gate; new prerequisite reads get fixtures + a validate-script check before their wave starts.
